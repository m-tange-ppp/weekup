# 通知重複送信の修正

**完了日**: 2026-05-19
**カテゴリ**: バグ修正

## 背景・原因

`scheduleAllNotifications` が「`cancelAllScheduledNotificationsAsync` → 再スケジュール」という2段階の非同期処理で構成されていた。この設計は非冪等であり、以下のシナリオで重複登録が発生しうる状態だった。

- バックグラウンド→フォアグラウンド復帰時の再マウント
- React 18 Strict Mode による `useEffect` の二重実行
- Android における identifier 上書き未保証

複数の `scheduleAllNotifications` 呼び出しが競合した場合、キャンセル後・再登録前に別の呼び出しが割り込み、最終的に各通知が2件ずつ登録されるというレースコンディションが発生していた。

## 対応方針

`expo-notifications` の `scheduleNotificationAsync` は **identifier をキーとした upsert** として機能する（iOS: `UNNotificationRequest`、Android: identifier ベースの AlarmManager）。そのため `cancelAllScheduledNotificationsAsync` による全削除は不要であり、直接 `scheduleNotificationAsync` を呼ぶだけで自然に冪等になる。

`notificationsEnabled=false` 時は `cancelAllScheduledNotificationsAsync`（他アプリ通知も消去するリスクあり）ではなく、`cancelScheduledNotificationAsync(id)` で WeekUp の3件のみを個別キャンセルする方式に変更した。

あわせて、未使用変数 `CHANNEL_ID` を削除した。

## 変更ファイル

| ファイル | 変更内容 |
| -------- | -------- |
| `services/NotificationService.ts` | `cancelAllNotifications` 関数を削除。`scheduleAllNotifications` から `cancelAllNotifications()` 呼び出しを除去し、`notificationsEnabled=false` 時は識別子ベースで3件を個別キャンセルに変更。未使用変数 `CHANNEL_ID` を削除。 |

## 受け入れ条件（確認済み）

- [x] `scheduleAllNotifications` を連続して複数回呼び出しても、スケジュール済み通知が3件以下であること
- [x] 通知をOFFに設定した後、スケジュール済み通知が0件であること
- [x] 通知をONに設定した後、スケジュール済み通知が3件（週始めリマインド・朝・夜）であること
- [x] TypeScript / ESLint エラーなし

## 備考

- `cancelAllScheduledNotificationsAsync` を廃止したことで、Android における他アプリ通知の誤消去リスクも排除した
- 呼び出し側（`app/_layout.tsx`・`hooks/use-settings.ts`）はシグネチャに変更がないため修正不要
